# Q106 — Stage 3 hypotheses (fixed before computation)

Four hypotheses on the catalog of design operations. Each operation is a named edit that moves the verdict
in a stated direction. The law's conditions give three levers — binding (the mediator reading a party),
liveness (the party keeping itself live), and requirement (all parties jointly needed rather than
substitutable) — and each lever carries a build operation (dyadic to triadic) and an inverse break
operation. Written and committed before the operations are run. The base forms (broadcast dyadic,
read_recipient triadic, one_shot dyadic) are the instrument control.

## H1 — Build operations build

- **Claim:** Each build operation — add binding, restore liveness, require all — takes its dyadic input to
  triadic.
- **H0:** Some build operation does not produce a triad.
- **Predicted outcome:** all build. H0 refuted.

## H2 — Break operations break

- **Claim:** Each break operation — remove binding, break liveness, substitute — takes its triadic input
  to dyadic.
- **H0:** Some break operation does not produce a dyad.
- **Predicted outcome:** all break. H0 refuted.

## H3 — Build and break are inverse pairs

- **Claim:** For each lever, the build operation's output form is the break operation's input form, and
  conversely — the two are inverses.
- **H0:** Some lever's build and break are not inverses.
- **Predicted outcome:** inverse pairs. H0 refuted.

## H4 — The operations partition into three levers

- **Claim:** The operations span exactly three levers — binding, liveness, requirement — each with one
  build and one break, matching the law's conditions.
- **H0:** The operations do not partition this way.
- **Predicted outcome:** three levers, balanced. H0 refuted. The design vocabulary is the law's conditions,
  read as moves.
